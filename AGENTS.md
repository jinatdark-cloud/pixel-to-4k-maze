# AGENTS.md

This file defines documentation standards for the **pixel-to-4k-maze** repository.

## Purpose
Maintain a coherent, serious, and scalable narrative and design language for a semi-open-world maze adventure game. All future docs should feel like parts of one long-term worldbuilding and production plan.

## Writing Tone and Voice
- Use clear, polished English.
- Keep tone **serious, mysterious, and layered**.
- Prefer grounded speculative worldbuilding over whimsical or comedic phrasing.
- Avoid childish expressions, parody, and generic high-fantasy clichés.
- Write with production intent: each document should be usable by design, narrative, and systems teams.

## Canon Terminology Rules
Use these terms consistently across all documentation:
- **The Maze**: the planetary-scale labyrinthine structure where exploration and advancement occur.
- **The Weave**: the underlying synthesis fabric that binds relics, resources, and progression systems.
- **Strata**: social layers that determine rights, access, and influence.
- **Access Tiers**: formal permissions required to enter deeper Maze regions.
- **Synthesis**: combining discovered components into tools, keys, upgrades, and pathways.
- **Expeditions**: player-led runs into Maze sectors for discovery, extraction, and advancement.
- **Anchors**: stable hubs (settlements/checkpoints) from which expeditions launch.

If introducing a new core term, define it once in context and then apply it consistently everywhere.

## Document Structure Conventions
For all major documentation files:
1. Start with a short purpose statement.
2. Use clear `##` and `###` headings in logical progression.
3. Move from high-level framing to practical implications.
4. Close with actionable design implications when relevant.

Recommended section order when applicable:
- **Overview / Purpose**
- **Core Concepts**
- **Systems or Social Implications**
- **Player Experience Impact**
- **Future Expansion Hooks**

## Consistency with README.md
- Keep all future documentation aligned with the current `README.md` vision, scope, and terminology.
- When `README.md` evolves, update documentation wording to remain consistent with it.
- If a conflict appears between a document and `README.md`, resolve the conflict in favor of `README.md` and note the adjustment in the edited document.

## Scope Discipline
- Focus on documentation only unless explicitly asked for implementation.
- Do not include code snippets in lore/design docs unless specifically requested.
- Keep mechanics practical enough for implementation, while preserving mystery in narrative framing.
